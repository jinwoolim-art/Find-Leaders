# Citizen Q&A Topic Classification — Backend Spec

**Repo:** `metarailix/ai-avatar-core` (branch `dev`)
**Author:** Tam (product) — drafted with Claude
**Status:** Draft for Sahab review
**Last updated:** 2026-05-20

---

## 1. Background — why this is needed

The Candidate Dashboard shows analytics such as **"Today's good answers TOP 3"**,
**"Answers to improve TOP 3"**, exposure counts, and topic-level satisfaction.

After auditing the current backend, **these cannot be produced from real data**, for
three structural reasons:

1. **No exposure tracking.** `chat_message_reactions` only stores a row when a
   citizen taps 👍/👎. A citizen who reads an answer and leaves produces no row.
   There is no impression/view counter anywhere. So "shown 142 times" is not a
   number the database can produce.

2. **No grouping key.** Every `chat_turns.assistantAnswer` is freshly generated by
   the LLM (RAG over `avatar_knowledge_chunk`). Two citizens asking the same thing
   get two different answer strings. There is nothing stable to aggregate on.

3. **The ranking query is structurally empty.** `analytics.service.getAnswerRanking`
   groups by `COALESCE("chatTurnId", md5("answerText"))`. `chatTurnId` is unique per
   turn → each group has at most 1 reaction → `HAVING COUNT(*) >= 10` always
   removes everything.

This spec introduces a **stable grouping key** — a canonical *topic* — so that
citizen interactions can be aggregated, counted, and ranked from real data.

It also unlocks a **secondary efficiency benefit** (Phase 2): once a question is
matched to a topic, a candidate-registered answer can be served instead of a fresh
LLM generation — faster, cheaper (fewer tokens), more consistent, and safer under
election law (the answer was reviewed by the candidate).

---

## 2. Goal

- Give every `chat_turns` row a nullable `matchedTopicId`.
- Make exposure count = number of `chat_turns` per topic (free, no extra counter).
- Make satisfaction / good / to-improve rankings aggregate by topic.
- Keep the change **additive and reversible** — no behavior change in Phase 1.

---

## 3. Phasing (risk control)

| Phase | Scope | Risk | Behavior change |
|-------|-------|------|-----------------|
| **Phase 1** | Topic catalog + classify each turn + analytics rewrite | Low — additive, nullable columns | None — chat answers unchanged |
| **Phase 1.5** | Safety & ops guards (§8 — A1–A5) | Low–Medium — read-only A1/A2, additive A3, pipeline guards A4/A5 | A1–A3: none; A4/A5: guards added to input handling & answer generation |
| **Phase 2** | Candidate-registered answers + serve them in the chat pipeline | Medium — changes how answers are produced | Yes — separate detailed spec |

**Phase 1 is the core deliverable of this document.** Phase 1.5 (§8) bundles
low-cost guards that ride on the topic structure. Phase 2 is sketched in §9 and
gets its own spec once Phase 1 is live.

## 4. Phase 1 — Data model

### 4.1 New table: `question_topic`

The canonical catalog of questions/topics. One row = one canonical question.
Scoped per election type (mayor / local council / superintendent).

```sql
CREATE TABLE "question_topic" (
  "id"              uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  "electionTypeId"  uuid NULL REFERENCES "election_types"("id") ON DELETE CASCADE,
  -- NULL electionTypeId = topic applies to every election type (shared base set)
  "category"        varchar(80)  NOT NULL,   -- e.g. '청년', '교통', '복지', '교육'
  "questionText"    text         NOT NULL,   -- canonical phrasing of the question
  "orderIndex"      int          NOT NULL DEFAULT 0,
  "isActive"        boolean      NOT NULL DEFAULT true,
  "embedding"       vector(1536) NULL,        -- centroid embedding of questionText
  "createdAt"       timestamptz  NOT NULL DEFAULT now(),
  "updatedAt"       timestamptz  NOT NULL DEFAULT now()
);

CREATE INDEX "IDX_question_topic_electionType_active"
  ON "question_topic" ("electionTypeId", "isActive");

CREATE INDEX "IDX_question_topic_embedding"
  ON "question_topic" USING ivfflat ("embedding" vector_cosine_ops)
  WITH (lists = 50);
```

Notes:
- `vector(1536)` matches `avatar_knowledge_chunk.embedding` — **use the same
  embedding model** as `embedding.embeddingModel` config. If that model's dimension
  is not 1536, change both to match.
- `embedding` is nullable so the table can be seeded first (text only) and
  embeddings filled by a one-time script (see §6.3).

### 4.2 Modify table: `chat_turns`

```sql
ALTER TABLE "chat_turns"
  ADD COLUMN "matchedTopicId"   uuid  NULL
    REFERENCES "question_topic"("id") ON DELETE SET NULL,
  ADD COLUMN "matchSimilarity"  real  NULL;

CREATE INDEX "IDX_chat_turns_matchedTopic_created"
  ON "chat_turns" ("matchedTopicId", "createdAt");
```

- `matchedTopicId` — the topic the citizen's question was classified to. `NULL`
  means "uncategorized" (no topic crossed the similarity threshold). The turn is
  still recorded normally; it just is not topic-attributed.
- `matchSimilarity` — cosine similarity of the best match (0–1). Kept for
  threshold tuning and diagnostics. Optional but cheap and very useful.

**Both columns are nullable.** Existing rows are untouched. Fully reversible.

## 5. Phase 1 — Classification logic

### 5.1 Where it runs

The chat orchestrator **already computes a query embedding for every citizen
message** for RAG retrieval:

- `realtime-orchestrator.service.ts` ≈ line 250
  `const embResult = await this.embeddingService.generateEmbeddingWithUsage(cleaned);`
  `queryEmbedding = embResult.embedding;`

Reuse this exact embedding. **No extra LLM or embedding call is needed.**

### 5.2 New service: `TopicMatchService`

```ts
// pseudo-code
async match(queryEmbedding: number[], electionTypeId: string | null):
    Promise<{ topicId: string; similarity: number } | null> {

  const literal = `[${queryEmbedding.join(',')}]`;
  const rows = await dataSource.query(`
    SELECT "id",
           1 - ("embedding" <=> $1::vector) AS similarity
    FROM "question_topic"
    WHERE "isActive" = true
      AND "embedding" IS NOT NULL
      AND ("electionTypeId" = $2 OR "electionTypeId" IS NULL)
    ORDER BY "embedding" <=> $1::vector
    LIMIT 1
  `, [literal, electionTypeId]);

  const top = rows[0];
  if (!top || top.similarity < SIMILARITY_THRESHOLD) return null;
  return { topicId: top.id, similarity: top.similarity };
}
```

- `<=>` = pgvector cosine distance; `1 - distance` = cosine similarity.
- `SIMILARITY_THRESHOLD` — config value. Suggested start **0.78**; tune with real
  traffic (see §10).
- Below threshold → return `null` → `matchedTopicId` stays `NULL`.
- `electionTypeId` for the candidate: resolve from
  `user.electionTypeId` of `chat_turns.candidateId`. The orchestrator already
  loads the avatar/candidate; pass it through.

### 5.3 Wiring into the turn payload

`matchedTopicId` and `matchSimilarity` must flow from the orchestrator to the
persist step:

1. **`ChatTurnCompletedPayload`** (`src/events/payloads/chat-turn.payload.ts`) —
   add two optional fields:
   ```ts
   matchedTopicId: string | null;
   matchSimilarity: number | null;
   ```
2. **`realtime-orchestrator.service.ts`** — call `TopicMatchService.match(...)`
   after the embedding is available, and set the two fields when building
   `turnPayload`. ⚠️ There are **three emit sites** — non-streaming, streaming,
   and cache-hit (semantic cache). **All three** must set the fields (cache-hit
   turns are real citizen questions and must count toward exposure).
3. **`debitTokenForChat`** (`payment.service.ts` ≈ line 177) — include
   `matchedTopicId` and `matchSimilarity` in `manager.create(ChatTurn, { ... })`.

If `TopicMatchService` throws or `queryEmbedding` is absent, both fields fall back
to `null` — the turn is still persisted. **Classification failure must never block
or fail a chat turn.**

## 6. Phase 1 — Migrations & seed

Follow the existing migration timestamp convention in `src/migrations/`.

### 6.1 Migration A — `CreateQuestionTopic`
Creates the `question_topic` table + indexes + IVFFlat index (§4.1).
Requires `CREATE EXTENSION IF NOT EXISTS vector` (already enabled by the knowledge
migration; safe to repeat).

### 6.2 Migration B — `AddMatchedTopicToChatTurns`
Adds `matchedTopicId`, `matchSimilarity`, FK, and index to `chat_turns` (§4.2).

### 6.3 Migration C — `SeedQuestionTopics`
Inserts the canonical topic rows (text + category + electionType + orderIndex).
- **Seed data is provided as `docs/question-topic-seed.json`** — 290 topics
  extracted from `question-bank.html` / `assets/question-bank.js`: 50 common
  (`electionType: null`) + 80 each for `headOfLocal`, `councilor`, and
  `superintendent`. See the file's `_meta` block for the field mapping.
- 10 of the 50 common topics are operator-answered (`audience: operator`); they
  are still real citizen questions, so they are kept as classifiable topics.
- The `electionType` key (`headOfLocal` / `councilor` / `superintendent`) must be
  resolved to an `election_types.id` at migration time — see §10.
- `embedding` is left `NULL` at seed time.
- A one-time script (or a guarded boot step) generates embeddings for every
  `question_topic` row with `embedding IS NULL`, using `EmbeddingService`. This is
  idempotent and can be re-run when topics are added.

### 6.4 Reversibility
- Every migration has a clean `down()`: drop columns / drop table.
- `matchedTopicId` is nullable → no data loss, no backfill required to ship.
- **Backfill of historical `chat_turns` is optional** — see §10.

## 7. Phase 1 — Analytics module impact

The analytics module on branch `feat/analytics-module` must be updated:

- **`getAnswerRanking` (best/worst)** — replace the broken
  `GROUP BY COALESCE("chatTurnId", md5("answerText"))` with
  `GROUP BY ct."matchedTopicId"`, joined to `question_topic` for the label.
  Exposure = `COUNT(ct.*)` per topic. Like ratio = reactions joined on
  `chatTurnId`. Keep `HAVING COUNT(ct.*) >= minExposures`.
- **`today` / `trend` / `satisfaction`** — unchanged in logic; may add an optional
  `WHERE ct."matchedTopicId" IS NOT NULL` for topic-scoped views.
- Keep the existing **cohort < 3 privacy guard**.
- UI labels: until topic data accumulates, "Today's good answers" should read as
  cumulative or show a graceful empty state — but that is a frontend change,
  out of scope here.

## 8. Phase 1.5 — Safety & ops guards

Phase 1.5 is a set of low-cost guards that **ride on the Phase 1 topic
structure**. None of them need an embedding or LLM call beyond what the chat
pipeline already does. They are listed here so Sahab can size them — each ships
independently and any can be deferred.

- **A1, A2 are read-only** analytics queries — zero pipeline risk.
- **A3** is additive schema (one nullable column + FK).
- **A4, A5** add guards inside the chat pipeline and do change input/answer
  handling within a narrow scope — flagged per item, both fail-open.

### A1 — Unanswered / weak popular topics

**What:** rank topics by exposure (`COUNT(chat_turns)`) and flag the ones asked
often but answered weakly, so the candidate knows where to focus.
**Phase 1 signal of "weak":** low like-ratio / low satisfaction for the topic.
**After Phase 2:** "weak" also covers topics with no `PUBLISHED`
`candidate_topic_answer` (§9) → "register an answer here".
**How:** a new analytics endpoint, e.g. `GET /candidate/analytics/topic-gaps`,
reusing the topic grouping from §7. No schema change.
**Risk:** none (read-only). **Behavior change:** none.

### A2 — Negative-reaction spike alert

**What:** when one topic accumulates a disproportionate share of 👎 (DISLIKE)
reactions in a recent window, surface "this answer is reacting badly".
**How:** a scheduled query (or computed on dashboard load) over
`chat_message_reactions` joined to `chat_turns.matchedTopicId`. A topic raises
the alert when its DISLIKE ratio exceeds a configurable threshold and exposure
meets the existing cohort-minimum guard. Delivery (in-dashboard badge vs. push)
is Tam/Sahab's call.
**Risk:** none (read-only). **Behavior change:** none.

### A3 — Citizen question → pledge linkage

**What:** connect topics to the candidate's structured pledges
(`candidate_profile_pledge`) so the dashboard can show "citizens asking about
topic X ↔ your pledge Y, and how it is reacting".
**How (cheap path):** embed each `candidate_profile_pledge` once — add a nullable
`embedding vector(1536)` column, filled by the same one-time script used for
topic embeddings (§6.3). In the orchestrator, match the **already-computed**
citizen query embedding against that candidate's pledges and store a nullable
`matchedPledgeId` on `chat_turns`. No extra embedding call for the citizen
question.
**Risk:** low — additive nullable columns + FK, reversible.
**Behavior change:** none — metadata only, answer generation untouched.
**Open:** confirm pledge count per candidate is small enough for a linear scan.

### A4 — PII masking on citizen input

**What:** when a citizen types personal data (phone number, email, resident
registration number) into a question, mask it **before** it is stored in
`chat_turns.userMessage` / `chat_message_reactions` and before it is sent to the
LLM.
**How:** a regex sanitizer applied where the orchestrator already produces the
`cleaned` input (§5.1). Reliable for phone / email / RRN patterns; free-form
street addresses are best-effort only and must not be promised as guaranteed.
**Risk:** low — additive, no schema change.
**Behavior change:** the stored and LLM-sent text is the masked version; the
citizen's on-screen experience is unchanged. Masking must never drop a turn —
on sanitizer error, store the text as-is.

### A5 — Hallucination guard + abuse filter

Two independent, fail-open guards in the chat pipeline:

**A5a — Hallucination guard.** When the RAG knowledge retrieval for a citizen
question returns only low-similarity chunks (the candidate's knowledge base does
not cover this area), instruct the LLM to **not invent a stance** — reply with
"the candidate has not stated a position on this yet" instead of a fabricated
answer. The retrieval similarity is already computed, so this costs nothing, and
it keeps the platform safer under election law.
*Note:* the signal is the **RAG knowledge retrieval** score (candidate
coverage), which is distinct from `matchSimilarity` (closeness to a standard
topic).
**Behavior change:** yes — but only in the low-coverage tail, gated by a
configurable threshold.

**A5b — Abuse / hate / obscenity filter.** Run the citizen input once through a
moderation check at intake (e.g. the OpenAI moderation API — free). If flagged,
handle it gracefully (decline politely; do not persist it as a normal turn).
**Behavior change:** flagged inputs are declined instead of answered.

**Risk (A5):** low–medium — pipeline guards, both configurable and fail-open: on
guard error, fall back to current behavior and never block a legitimate turn.

### Suggested ordering

A1 + A2 first (read-only, zero risk, immediate dashboard value) → A3 (additive
metadata) → A4 + A5 (pipeline guards, need the most review). A4 and A5 can be
moved into their own mini-spec if Sahab prefers to keep this round purely
additive.

## 9. Phase 2 — Candidate-registered answers (sketch, separate spec later)

Once topics exist, candidates can register a reviewed answer per topic:

```sql
CREATE TABLE "candidate_topic_answer" (
  "id"           uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  "candidateId"  uuid NOT NULL REFERENCES "user"("id") ON DELETE CASCADE,
  "topicId"      uuid NOT NULL REFERENCES "question_topic"("id") ON DELETE CASCADE,
  "answerText"   text NOT NULL,
  "status"       varchar(16) NOT NULL DEFAULT 'DRAFT',  -- DRAFT | PUBLISHED
  "createdAt"    timestamptz NOT NULL DEFAULT now(),
  "updatedAt"    timestamptz NOT NULL DEFAULT now(),
  UNIQUE ("candidateId", "topicId")
);
```

Chat pipeline change: when `matchedTopicId` resolves AND a `PUBLISHED`
`candidate_topic_answer` exists → serve / prioritize that answer instead of a
fresh RAG generation. This is a **behavior change** and needs its own spec
(fallback rules, partial coverage, tone blending, caching). Not in this round.
